# 编程学习沉淀系统

为非技术背景的人设计的学习文档体系。核心理念：**"怎么做"和"是什么"分开放，每天自动沉淀。**

## 文档架构

```
learning-docs/
├── index.md              ← 导航入口
├── playbook.md           ← 实操手册（SOP + 提示词 + 命令速查）
├── knowledge.md          ← 知识库（概念按主题沉淀）
├── learning-log.md       ← 每日学习速记（最新在最上面）
├── open-questions.md     ← 待解决问题
└── projects/             ← 项目研究笔记
    └── _template.md      ← 研究新项目时复制此模板
```

| 文件 | 干什么用 | 什么时候翻 |
|------|---------|-----------|
| playbook.md | SOP、提示词、命令速查 | 做事时照着走 |
| knowledge.md | 概念按主题沉淀 | 碰到不懂的词回来查 |
| learning-log.md | 每日学习速记 | 每天学完速记 |
| open-questions.md | 还没搞懂的问题 | 学习中随时记 |
| projects/ | 每个项目一个文件 | 研究新项目时 |

## 首次搭建

用户首次触发且学习文档不存在时：

1. 询问存放位置（建议 `~/Desktop/learning-docs/`）
2. 用 [templates.md](references/templates.md) 中的模板创建全套文件
3. 建议安装配套 Cursor Rule——见 [companion-rules.md](references/companion-rules.md)

## 内容分流规则

用户说"记一下""加到笔记""帮我整理"时，判断内容类型写入对应文件：

| 内容类型 | → 文件 | 判断信号 |
|---------|--------|---------|
| 操作流程 / SOP | `playbook.md` | "怎么做……"、工作流 |
| 提示词 / 提问模板 | `playbook.md` | 给 AI 用的提示词 |
| 终端命令 | `playbook.md` 命令速查区 | git、npm、conda 等 |
| 概念定义 | `knowledge.md` | "什么是 X？" |
| 工具/机制理解 | `knowledge.md` | "X 怎么工作" |
| 未解决的问题 | `open-questions.md` | "为什么……？" |
| 项目研究 | `projects/[项目名].md` | 研究别人的项目 |

**Playbook vs Knowledge 判断标准：** 直接指导"下一步该做什么"的（即使含概念解释）→ Playbook；纯定义、不指导行动的 → Knowledge。边界模糊优先放 Playbook——翻一个地方比翻两个方便。

## 内容格式规范

### SOP（写入 playbook.md）

- 一个 SOP = 一个完整场景，有明确起点和终点
- 同一主线的多个 SOP → 合并为一个，Step 1/2/3 分步
- 轻量变体 → 标注"到这里可以停"，不独立成 SOP
- 步骤编号用 Step 1/2/3 或第 1/2/3 轮，保持简洁。避免用 Phase 0 / 0.5 / 1.5 等会越加越碎的编号
- 每步 = **可复制的提示词或命令**（`>` 引用块）+ 一句话目的
- 不写大段解释——保持行动导向
- 能一张表说清的不拆两张——如"判断标志 + 类型 + 用法"应在同一行看完，而不是一张表讲判断、另一张讲用法
- 每个 SOP 末尾附**速查卡片**（步骤 | 做什么 | 模式）
- 增补时审视周边冗余，该合并就合并

### 概念（写入 knowledge.md）

- 按关联性分组，每组标题点明**核心问题**
- 每个概念一行：**名称 | 一句话定义 | 生活类比**
- 关系图用 ASCII（`→ ↓ ├ └`），不用 Mermaid
- 易混淆的概念对 → 加对比表
- 不跨组重复，用"见概念组 X"交叉引用

### 命令（写入 playbook.md 命令速查区）

- 按**使用场景**分组（不按工具分组）
- `bash` 代码块 + 行内注释
- 危险操作标注"（危险）"，安全只读标注"（安全只读）"

### 新概念格式

每个新概念必须有：**一句话定义 + 生活类比**。命令必须完整可复制。

### 格式约束

- 不用 emoji（除非用户原始笔记里有且表意明确的保留）

## 收工日志沉淀

**触发词：** "收工""今天到这""结束吧""done for today""wrap up" 或类似表达。

### 第 1 步：提取知识点

**记录：**
- 第一次理解的新概念
- 认知升级（对已知事物更深的理解）
- 新学会的思维方式或方法论
- 重要的工具/机制理解

**不记录：**
- 具体操作步骤（建文件、改配置等）
- 反复试错过程
- 发散闲聊
- 对已知概念的重复确认

### 第 2 步：归类整理

- 同主题合并一条（Git 相关 → 一条 **Git**）
- 每条**加粗主题词**开头 + 要点
- 排序：新概念 → 深度讨论 → 小知识点
- 每条简洁一句话，不做多余的否定式补充说明

### 第 3 步：用户确认

列出清单等用户确认。用户可能删改或补充。

### 第 4 步：写入 learning-log.md

```
## YYYY-MM-DD

- 知识点 1
- 知识点 2
```

- 最新日期在最上面
- 当天已有 → 增量追加，不重复
- 一天可多次收工，每次只写本轮新增

## 整理用户笔记

用户分享未整理笔记或说"帮我整理笔记"时：

1. 通读，识别与已有文档的重叠
2. 重叠 → **合并**进已有内容，不新建重复条目
3. 全新 → 分类：SOP / 概念 / 命令
4. **保留用户自己的洞察和类比**——个人理解比教科书定义更有价值
5. 去掉对话语气和重复论述，保留所有命令和提示词原文
6. 整理完报告：合并了什么、新增了什么、去掉了什么

## 沟通方式

用户是非技术背景：

- **自顶而下**：先说"是什么、干什么用"，再说"怎么实现"
- 复杂概念用**生活化或产品化类比**
- 展示代码时简要说明含义，帮助建立代码感觉
- 优先补**工程常识**而非语法细节
- 复杂流程画 ASCII 流程图
- 主动教实用的 **AI 提问技巧**
- 概念有重要关联时主动提醒，帮构建知识网络

## 上下文预警

观察到上下文使用超过 ~70% 时主动提醒："上下文快满了，建议先收工沉淀日志，再开新对话继续。"
